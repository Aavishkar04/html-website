What It Does  
DevLens is a web application where developers paste or upload their code and receive a detailed  
AI-powered code review — not just generic suggestions, but specific feedback on: code quality,  
potential bugs, security vulnerabilities, performance issues, and best practice violations. The review is  
structured like a real senior engineer's review, not a list of ChatGPT bullet points.  
What separates this from 'just calling the OpenAI API': the application implements a structured review  
pipeline — it first classifies the language and context, then runs the code through multiple specialized  
review prompts (security lens, performance lens, readability lens), and finally synthesizes a coherent  
review with severity ratings. This multi-stage pipeline architecture is what makes it technically  
impressive.

What Makes This Architecturally Impressive  
• You implement prompt chaining — an advanced LLM pattern where the output of one prompt  
becomes the input of the next. This is what real AI products do.  
• Streaming responses — instead of waiting 10 seconds for the full review, you stream tokens as they  
arrive (like ChatGPT does). This requires understanding Node.js streams and React state updates.

rate limit errors. These are what separate a prototype from a real product.

What It Does  
RealSync is a real-time collaborative whiteboard where multiple users can draw, add shapes, write text,  
and see each other's cursors simultaneously — like a stripped-down Miro or FigJam. Users create a  
board, share a link, and anyone with the link can join and collaborate instantly.  
Why this is technically hard: real-time collaboration requires solving problems that CRUD applications  
never encounter — what happens when two users edit the same element simultaneously? What  
happens when a user disconnects and reconnects? How do you sync board state to a new user who  
joins mid-session? These are the problems that distinguish backend engineers who understand  
distributed systems from those who don't.  
Why This Project Gets You Hired  
WebSockets and real-time systems are among the most common advanced interview topics at product  
companies. Every company building chat, notifications, live dashboards, or collaborative features  
needs engineers who understand this. Showing that you built a working real-time system — not just  
read about it — puts you in a tiny minority of junior candidates.

The Genuinely Hard Parts (And How to Solve Them)  
• Canvas coordinate system: Canvas uses its own coordinate system that must be mapped to React  
state. When elements are moved or the canvas is resized, you must recalculate positions. Use a  
transform matrix.  
• Cursor smoothing: Raw cursor events come in at 60fps but Socket.io can't send 60 events/second  
per user. Solution: throttle cursor events to 30fps on client, interpolate on receiver side for smooth  
movement.  
• State explosion: 10 users drawing simultaneously = 10 × 60 events/second = 600 events/second.  
Solution: batch events — aggregate 50ms of drawing into one payload before emitting.  
• The reconnection problem: User disconnects, misses 200 events, reconnects. Solution: server  
keeps event log with sequence numbers. On reconnect, client sends its last sequence number,  
server sends all events after that.

What It Does  
HireTrack India is a full-featured job application tracker built specifically for Indian software developers.  
Users log every job application, track its status through the pipeline, log interview questions and  
outcomes, and get AI-powered insights about their job search — what's working, what's not, which  
companies respond to which types of profiles, and what the market is looking like based on aggregated  
(anonymous) data from other users on the platform.  
The Indian developer job market has no good tool for this. Internationally, tools like Huntr and Teal exist  
but they are built for Western markets — wrong company names, wrong platforms, wrong salary  
formats, wrong interview structures. This is a tool built by an Indian developer who is actively going  
through the job search, for other Indian developers doing the same. That authenticity is a product moat.  
WHY THIS HAS REAL 5,000+ USER POTENTIAL  
India produces ~1.5 million engineering graduates per year. Of those, a significant portion are  
actively job-seeking at any given time.  
The Indian developer community is highly active on Twitter/X, LinkedIn, and communities like  
DevsHQ, GeeksforGeeks, CodeChef.  
A tool built specifically for Naukri, LinkedIn, Instahyre, Wellfound, Cutshort — the platforms they  
actually use — with INR salary tracking and Indian company-specific data will spread organically.  
You are building this while going through the exact same job search. You ARE the target user.

The 5 Properties of an Impressive Project  
After reviewing what causes recruiters to stop and engage with a project versus scroll past, five  
properties consistently appear. All three projects above have all five.  
1. Solves a Real Problem  
Not 'I wanted to learn React so I built a todo app.' But 'I was having this specific problem and built  
this to solve it.' DevLens: developers want code review but can't always get it. RealSync: teams  
need quick collaboration without paying for Miro. HireTrack: Indian devs have no tool built for their  
market.  
2. Has Technical Depth That Takes More Than One Interview to Exhaust  
A todo app can be discussed in 3 minutes. DevLens's prompt pipeline, RealSync's conflict  
resolution, HireTrack's community analytics — these can each sustain 45 minutes of technical  
discussion. You want the interviewer to run out of time before they run out of questions.  
3. Is Live and Accessible (Not 'Here's the Code')  
Every project in this document is deployed with a real domain. The interviewer can open it on their  
phone during the interview. This removes all doubt about whether it actually works and  
demonstrates you know how to ship, not just code.  
4. Has a User (Even If It's Just You)  
A project with zero users is a demo. A project with 50 users is a product. HireTrack with 3,000  
users is something you talk about differently — with confidence, with data, with specific things you  
learned from real usage. Aim for users before you apply for jobs, not after.  
5. You Can Talk About What You Would Do Next  
'Given more time I would add X, Y, Z because users told me they need it' shows product thinking. 'I  
would scale it by doing A, B, C' shows engineering maturity. Every interviewer asks 'what would  
you improve?' Have 5 specific, non-obvious answers ready.

FINAL WORD — WHAT SEPARATES BUILDERS FROM WATCHERS  
There are two kinds of developers who attempt a portfolio transition: those who watch tutorials about  
projects and those who build them. The tutorial watchers feel productive. The builders feel frustrated,  
stuck, and uncertain — and they are the ones who get hired.  
Every feature described in these three projects will, at some point, seem impossible. The OAuth flow  
will break in a way the documentation doesn't cover. The Socket.io events will fire twice and you won't  
know why. The OpenAI API will return malformed JSON under token pressure. These are not failures  
— they are the actual job. Every senior engineer you admire spent hours on problems exactly like these  
before they understood them.  
The difference between a project that gets you hired and one that doesn't is not the idea. It is the level  
of finish. An 80% complete ambitious project looks like an incomplete project. A 100% complete  
ambitious project — deployed, documented, usable by strangers, with a README that shows how you  
think — looks like a product. Finish it.
